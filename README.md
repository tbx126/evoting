# Blockchain E-Voting Prototype

Browser-to-chain electronic voting prototype built with Solidity, Hardhat, Circom, snarkjs, Python, and static HTML clients.

The project combines three mechanisms in one workflow:

- Groth16 zero-knowledge proofs for vote legality and tally correctness
- BabyJubJub ElGamal for confidential one-hot ballots and homomorphic tallying
- Merkle-based receipt verification for post-election auditability

This repository is a research prototype, not a production election system.

## Features

- On-chain election lifecycle: `Created -> Active -> Ended -> Tallied`
- Integrated voter registration and one-vote enforcement in `contracts/Voting.sol`
- Vote submission with Groth16 proof verification in `castVote`
- Homomorphic tally submission with Groth16 proof verification in `updateTallyResults`
- Voter receipt storage in the browser and anonymous Merkle inclusion verification
- Hardhat integration tests and Python cryptographic unit tests

## Tech Stack

- Smart contracts: Solidity + Hardhat
- ZK tooling: Circom + snarkjs
- Cryptography: BabyJubJub ElGamal, Poseidon, Merkle proofs
- Frontend: static HTML + browser-side JavaScript
- Local server: FastAPI + Uvicorn
- Crypto validation: Python + pytest

## Prerequisites

- Node.js and npm
- Python 3.10+
- A local Hardhat node for local demo flow
- The vendored browser bundle `frontend/lib/circomlibjs.bundle.js` tracked in this repository
- ZK proof artifacts provisioned under `frontend/zk/`

Required proof artifacts:

- `frontend/zk/vote_proof.wasm`
- `frontend/zk/vote_proof_final.zkey`
- `frontend/zk/tally_proof.wasm`
- `frontend/zk/tally_proof_final.zkey`

These files are expected by both the frontend flow and the Hardhat test suite, but are not tracked by Git in this repository.

## Local Development

Start three terminals.

### 1) Start the local chain

```bash
npm run node
```

### 2) Deploy contracts and generate frontend config

```bash
npm run deploy:local
```

Local deployment also:

- deploys `VoteVerifier`, `TallyVerifier`, and `Voting`
- preloads two candidates
- registers the first five Hardhat accounts as voters
- generates `frontend/config.js`
- auto-generates a local ElGamal admin secret key when `ELGAMAL_SK` is not set

On the admin page, the local bootstrap key is displayed for convenience, but it is not auto-loaded. The administrator must still explicitly load the key before tallying.

### 3) Start the static frontend server

```bash
npm run server
```

Open:

- Voter UI: `http://localhost:8000/`
- Admin UI: `http://localhost:8000/admin`

## Deployment Notes

### Sepolia

Before deploying to Sepolia, set environment variables in `.env`.

Option A: provide the ElGamal secret key directly.

```env
SEPOLIA_RPC_URL=...
PRIVATE_KEY=...
ELGAMAL_SK=...
```

Option B: provide a precomputed ElGamal public key instead.

```env
SEPOLIA_RPC_URL=...
PRIVATE_KEY=...
ELGAMAL_PK_X=...
ELGAMAL_PK_Y=...
```

Then run:

```bash
npm run deploy:sepolia
```

For non-local deployments, `scripts/deploy.js` requires either `ELGAMAL_SK` or both `ELGAMAL_PK_X` and `ELGAMAL_PK_Y`. There is no insecure fallback secret key anymore.

If you deploy with only `ELGAMAL_PK_X` and `ELGAMAL_PK_Y`, the corresponding ElGamal secret key remains off-chain and must still be loaded manually in `frontend/admin.html` before tallying.

## Implementation Notes

- The current circuit/contract configuration is fixed to `N = 2` candidates.
- `frontend/config.js` is generated by `scripts/deploy.js` and is intentionally not tracked.
- `frontend/lib/circomlibjs.bundle.js` is tracked so fresh clones can open the frontend without rebuilding the browser bundle first.
- Encrypted vote payloads are emitted as events, while commitment and ciphertext hash are stored on-chain.
- The voter verification flow checks the audit bundle root against on-chain `merkleRoot` before validating a Merkle proof.

## Known Limitations

- Fixed two-candidate circuit artifacts in the current setup
- External ZK artifact provisioning is required
- Single-admin ElGamal key custody remains a trust assumption
- Not designed for coercion resistance or production governance
