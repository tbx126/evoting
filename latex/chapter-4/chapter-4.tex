\section{Project Architecture}

The repository is organized into five major modules:
\begin{itemize}
    \item \texttt{contracts/}: Solidity contracts for election control (including integrated voter registration) and verifier adapters.
    \item \texttt{circuits/}: Circom circuits for vote and tally proofs.
    \item \texttt{frontend/}: voter and administrator web clients, including browser-side cryptographic helpers.
    \item \texttt{crypto/}: Python cryptographic implementation for BabyJubJub ElGamal and tally utilities.
    \item \texttt{test/} and \texttt{tests/}: JavaScript/Hardhat and Python/Pytest verification suites.
\end{itemize}

This modularity allows independent testing of smart-contract logic, cryptographic primitives, and user-facing flows.

\section{Smart Contract Design}

\subsection{Core Contract Responsibilities}

The voting contract is responsible for:
\begin{itemize}
    \item election initialization and candidate registration;
    \item lifecycle transition enforcement;
    \item vote acceptance conditioned on proof verification and voter eligibility;
    \item tally finalization conditioned on proof verification and consistency checks;
    \item Merkle-root publication for post-election audit.
\end{itemize}

Voter registration and one-vote enforcement are handled directly within the Voting contract, eliminating a separate registry dependency. Verifier contracts expose on-chain Groth16 verification interfaces.

\subsection{On-Chain Data Model}

Key on-chain structures include candidate records, per-voter vote metadata (commitment and ciphertext hash), a global commitment array, and persisted point-form tally outputs (\texttt{tallyResultPointX}/\texttt{tallyResultPointY}). Ciphertext payload itself is emitted as event data to reduce storage cost while preserving off-chain reconstructability.

\subsection{Lifecycle Safeguards}

Contract modifiers enforce role and state restrictions. Invalid phase transitions, unauthorized calls, duplicate votes, and malformed proof submissions are rejected before state mutation. One deliberate exception is \texttt{updateMerkleRoot}: it remains admin-callable until status \texttt{Tallied}. This governance tradeoff and its assurance impact are discussed in Section 5.8 (P2: Merkle root mutability).

\section{Circuit Implementation}

\subsection{Vote Circuit}

The vote circuit integrates Poseidon commitment checks, candidate range checks, one-hot encryption consistency checks, and ciphertext-hash binding. Candidate range enforcement is implemented with \texttt{LessThan(8)}, so candidate indices are bounded to at most 255. In this repository, the instantiated configuration is fixed to \texttt{N=2}.

\subsection{Tally Circuit}

The tally circuit verifies key ownership relation and per-candidate decryption correctness for aggregated ciphertexts. For total consistency, it enforces equality at the curve-point level: the sum of public result points must equal \texttt{totalVotes} $\cdot G$. This is implemented with \texttt{Num2Bits(32)} for \texttt{totalVotes} encoding, which implies an upper bound of approximately $2^{32}-1$ on directly representable totals in-circuit. Public signals are exported and consumed by the Solidity verifier. Because consistency is point-based, strict integer-to-point binding is not fully enforced by the circuit alone; it would require additional contract-level constraints that remain unimplemented (see Section 5.8 for the corresponding P1 gap analysis).

\subsection{Artifact Pipeline}

Proof generation follows the standard Groth16 flow: circuit compilation, trusted setup artifacts, witness construction, proof generation, and verifier calldata export \cite{circomdocs,snarkjs}. Contract-side verification executes with precompiled pairing checks.

\section{Frontend Engineering Workflow}

\subsection{Voter Interface}

The voter page executes the following sequence:
\begin{enumerate}
    \item initialize cryptographic runtime and contract connections;
    \item fetch candidate list and election state;
    \item on candidate selection, generate ciphertexts and commitment;
    \item generate vote proof and prepare calldata;
    \item submit transaction through wallet;
    \item persist local receipt for later anonymous verification.
\end{enumerate}

\subsection{Administrator Interface}

The administrator page performs:
\begin{enumerate}
    \item encrypted vote event retrieval;
    \item ciphertext reconstruction and integrity filtering by recomputing Poseidon \texttt{ciphertextHash} (mismatched entries are dropped);
    \item homomorphic aggregation, followed by decryption and candidate result extraction;
    \item tally proof generation (browser-side WASM in current implementation) and contract submission;
    \item deterministic commitment Merkle-tree construction after sorting by \texttt{blockNumber}, then \texttt{transactionIndex}, then \texttt{logIndex};
    \item audit-bundle export and local cache.
\end{enumerate}

\subsection{Audit Utility Module}

The audit utility in \texttt{frontend/lib/audit.js} centralizes:
\begin{itemize}
    \item bytes32 normalization;
    \item Merkle tree construction with duplicate-last odd handling;
    \item Merkle inclusion proof generation and path verification;
    \item election-key namespace generation for local storage isolation.
\end{itemize}

This separation reduces logic duplication and improves testability.

\section{Testing and Validation Strategy}

\subsection{Contract-Level Tests}

Hardhat tests cover role control, lifecycle transitions, valid/invalid voting attempts, and tally submission behavior with Groth16 proofs. End-to-end scenarios include realistic proof generation and verifier invocation. To keep suite runtime stable, expensive Groth16 proofs are generated once in a shared setup hook and then reused across behavior-focused test cases.

\subsection{Cryptographic Unit Tests}

Pytest cases validate curve arithmetic invariants, key generation, encryption/decryption correctness, homomorphic addition, one-hot vote encoding semantics, and bounded discrete-log solving.

\subsection{Engineering Observations}

The dual test stack provides coverage at both the contract protocol layer and the cryptographic primitive layer, giving high confidence in isolated module correctness. However, integration testing revealed several boundary conditions worth noting for future development.

First, the most brittle integration point is the mapping between circuit public signals and Solidity function parameters. Because snarkjs exports proof calldata in a specific encoding, any mismatch in signal ordering between the circuit definition and the contract's expected \texttt{pubSignals} array causes a generic rejection rather than a descriptive per-signal error. This was addressed by careful alignment during circuit compilation, but requires maintenance discipline whenever circuits are recompiled.

Second, browser-side proof generation is resource-intensive. During development, memory pressure was observed on some low-memory devices and caused WASM instability. The voter interface should therefore include progress indicators and graceful error messages for proof generation failures.

Third, ElGamal tally decryption relies on bounded runtime discrete-log recovery (baby-step giant-step). In the current administrator workflow, the search bound is set dynamically to \texttt{totalVotes + 100}. If recovery falls outside this bound, the client raises an explicit error. Since each per-candidate tally is upper-bounded by \texttt{totalVotes}, this bound also upper-bounds per-candidate recovery in practice. The selected bound should still be documented and reviewed for larger-election deployment profiles.

\section{Deployment Notes}

Local deployment uses a Hardhat node and a Python-served static frontend. The deployment script (\texttt{scripts/deploy.js}) generates \texttt{frontend/config.js} automatically, which is required before the frontend pages can interact with the deployed contracts.

For testnet or mainnet deployment, the following additional steps apply:
\begin{itemize}
    \item Generate and securely store an ElGamal keypair before deployment. Pass \texttt{PK} into the Voting constructor, and keep \texttt{sk} outside the contract and off persistent storage; load it into the administrator client only at tally time.
    \item Configure \texttt{.env} with the deployer private key and the target network RPC URL before running the deploy script.
    \item Ensure the \texttt{frontend/zk/} directory contains the compiled WASM witness generators and proving key files (\texttt{.zkey}) corresponding to the deployed verifier contracts.
    \item Configure MetaMask in the voter and administrator browsers to point to the correct network (chain ID must match the deployment target).
    \item The trusted setup lineage (the \texttt{.ptau} used for key generation and circuit-specific \texttt{.zkey} files) must be kept consistent with the deployed verifier contracts. Recompiling circuits without redeploying verifiers will cause all proof submissions to fail.
\end{itemize}

These operational dependencies are documented in the repository README and deployment scripts.
