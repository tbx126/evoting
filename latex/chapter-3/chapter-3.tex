\section{Protocol Roles}

The protocol operates through three active roles. An administrator deploys the contracts at the outset. This actor then maintains the voter whitelist and retains sole possession of the private key that decrypts the final tally. Registration as a voter hinges on one requirement alone: the address must appear on the whitelist. That condition grants the right to submit exactly one ballot. A public verifier is any party that can read blockchain data without credentials: they can observe on-chain state transitions, inspect proof-verification outcomes, and run the audit procedure in Section~3.9 using the published audit bundle. Above all, the system secures ballot confidentiality, guarantees result correctness, and preserves audit trace integrity.

\section{Threat Model}

The model considers malicious voters and passive blockchain observers. For administrator behavior, the prototype assumes a \emph{semi-trusted administrator}: the administrator is expected to follow the protocol, while single-key control remains an explicit trust assumption rather than a cryptographically enforced guarantee. The following attacks are explicitly targeted:
\begin{itemize}
    \item invalid ballot shape (non-one-hot encryption);
    \item out-of-range candidate encoding;
    \item replay or duplicate voting attempts;
    \item fabricated tally claims unsupported by valid proof.
\end{itemize}

The following are not fully solved in the current prototype: coercion resistance, malware-compromised browser clients, threshold key governance, fully adversarial administrator behavior (for example refusing to tally or submitting inconsistent off-chain artifacts), and per-candidate integer-to-result-point binding at contract layer (ZKP3 constrains curve-point correctness, while integer mapping remains client-verifiable rather than contract-enforced). \section{Protocol State Machine}

Election lifecycle states are:
\begin{equation}
S = \{\texttt{Created},\ \texttt{Active},\ \texttt{Ended},\ \texttt{Tallied}\}
\end{equation}

Allowed transitions are:
\begin{equation}
\texttt{Created} \rightarrow \texttt{Active} \rightarrow \texttt{Ended} \rightarrow \texttt{Tallied}
\end{equation}

This deterministic state machine ensures that vote submission only occurs in the active phase and tally finalization only occurs after the election is ended.

\section{Ballot Encoding and Encryption}

Let $N$ be the number of candidates and $\mathit{candidateId} \in \{0,1,\dots,N-1\}$ be the selected candidate index. A ballot is encoded as one-hot vector
$\mathbf{m}=(m_0,\dots,m_{N-1})$ where $m_{\mathit{candidateId}}=1$ and all other entries are $0$.

For each candidate position $j$, ElGamal ciphertext is generated on BabyJubJub:
\begin{equation}
C1_j = r_j \cdot G, \quad C2_j = m_j \cdot G + r_j \cdot \mathit{PK}
\end{equation}
where $r_j$ is sampled uniformly from the BabyJubJub scalar field, $G$ is generator, and $\mathit{PK}$ is administrator public key. We write $C1_j=(\mathit{c1X}_j,\mathit{c1Y}_j)$ and $C2_j=(\mathit{c2X}_j,\mathit{c2Y}_j)$ for affine coordinates, which are used by the vote-proof witness and ciphertext binding hash. The notation is general in $N$, while the current deployed prototype instantiates $N=2$ and enforces this cardinality at contract layer.

\section{Commitment and Ciphertext Binding}

For private random salt $\mathit{salt}$, commitment is defined as:
\begin{equation}
\mathit{commitment} = \mathsf{Poseidon}(\mathit{candidateId},\, \mathit{salt})
\end{equation}

Ciphertext binding hash is defined over all ciphertext coordinates:
\begin{equation}
\begin{split}
H_{ct} = \mathsf{Poseidon}(&\mathit{c1X}_0,\,\mathit{c1Y}_0,\,\mathit{c2X}_0,\,\mathit{c2Y}_0,\,\dots,\\
                  &\mathit{c1X}_{N-1},\,\mathit{c1Y}_{N-1},\,\mathit{c2X}_{N-1},\,\mathit{c2Y}_{N-1})
\end{split}
\end{equation}

The vote proof uses $(\mathit{commitment}, H_{ct}, \mathit{PK}_x, \mathit{PK}_y)$ as public signals and proves consistency between commitment, candidate index, and encrypted one-hot structure.
Here, $\mathit{PK}_x, \mathit{PK}_y$ denote affine coordinates of $\mathit{PK}$.
In the reference tally client, each emitted ciphertext payload is re-hashed off-chain and matched against the on-chain vote record $(\mathit{commitment}, \mathit{ciphertextHash})$ before aggregation; mismatched entries are excluded from the tally set. This is a client-side integrity requirement rather than a contract-enforced invariant.

\section{Vote Proof Constraints}

The vote circuit takes private inputs $(\mathit{candidateId},\, \mathit{salt},\, r_j,\, \mathit{c1X}_j,\, \mathit{c1Y}_j,\, \mathit{c2X}_j,\, \mathit{c2Y}_j)$ for $j \in \{0,\dots,N{-}1\}$, and exposes four public signals $(\mathit{commitment},\, H_{ct},\, \mathit{PK}_x,\, \mathit{PK}_y)$. It enforces the following constraint families.
\begin{enumerate}
    \item \textbf{Commitment consistency.} The circuit recomputes $\mathsf{Poseidon}(\mathit{candidateId},\,\mathit{salt})$ from private inputs and constrains the result to equal the public signal $\mathit{commitment}$. This binds the on-chain commitment to a specific candidate choice without disclosing it.
    \item \textbf{Candidate range.} The circuit asserts $0 \le \mathit{candidateId} < N$, where $N$ is the candidate cardinality hardcoded at compilation time. Any index outside this range causes the constraint system to be unsatisfiable.
    \item \textbf{One-hot encryption.} For each slot $j \in \{0,\dots,N{-}1\}$, the circuit verifies that the ElGamal ciphertext encodes $m_j = 1$ when $j = \mathit{candidateId}$ and $m_j = 0$ otherwise. Together with the range constraint, this guarantees that every accepted ballot assigns exactly one vote to exactly one candidate.
    \item \textbf{Ciphertext binding.} The circuit hashes all $4N$ ciphertext coordinates $(\mathit{c1X}_0,\,\mathit{c1Y}_0,\,\mathit{c2X}_0,\,\mathit{c2Y}_0,\,\dots,\,\mathit{c1X}_{N-1},\,\mathit{c1Y}_{N-1},\,\mathit{c2X}_{N-1},\,\mathit{c2Y}_{N-1})$ with Poseidon and constrains the digest to equal the public signal $H_{ct}$. Because $H_{ct}$ is recorded on-chain alongside the commitment, any post-proof substitution of ciphertext components is detectable.
\end{enumerate}

When the proof is accepted by the on-chain Groth16 verifier, these four constraints are satisfied simultaneously. The contract therefore treats the accompanying ballot as a well-formed, in-range, single-candidate vote whose encrypted payload is irrevocably bound to the published commitment.

\section{Homomorphic Tally and Tally Proof}

For each candidate position $j$, the aggregated ciphertext is computed by elliptic-curve point addition over all $V$ cast ballots:
\begin{equation}
\bar{C1}_j = \sum_{k=1}^{V} C1_{k,j}, \quad \bar{C2}_j = \sum_{k=1}^{V} C2_{k,j}
\end{equation}

Given the administrator secret key $\mathit{sk}$, the decryption result point for each candidate is:
\begin{equation}
R_j = \bar{C2}_j - \mathit{sk} \cdot \bar{C1}_j
\end{equation}

By the homomorphic property of ElGamal encryption, each $R_j$ equals $n_j \cdot G$, where $n_j$ is the integer vote count for candidate $j$. The administrator recovers $n_j$ off-chain via bounded discrete-log search and submits the integer results to the contract.

The tally circuit takes a single private input $\mathit{sk}$ and exposes all remaining values as public signals: $(\mathit{PK}_x,\, \mathit{PK}_y,\, \bar{C1}^{X}_j,\, \bar{C1}^{Y}_j,\, \bar{C2}^{X}_j,\, \bar{C2}^{Y}_j,\, R^{X}_j,\, R^{Y}_j,\, \mathit{totalVotes})$ for $j \in \{0,\dots,N{-}1\}$. It enforces three constraint families.
\begin{enumerate}
    \item \textbf{Key ownership.} $\mathit{PK} = \mathit{sk} \cdot G$, confirming that the prover holds the secret key corresponding to the election public key.
    \item \textbf{Decryption correctness.} For each candidate $j$, the circuit verifies $R_j = \bar{C2}_j - \mathit{sk} \cdot \bar{C1}_j$, ensuring that each result point is the honest decryption of the aggregated ciphertext.
    \item \textbf{Sum consistency.} The circuit checks $\sum_{j=0}^{N-1} R_j = \mathit{totalVotes} \cdot G$ in curve-point arithmetic, confirming that the individual result points account for the declared total.
\end{enumerate}

Before invoking the Groth16 verifier, the contract performs two pre-checks: the public-key coordinates in the proof's public signals must match the on-chain election key, and the $\mathit{totalVotes}$ signal must equal the contract's internal vote counter. If the proof is accepted, the contract stores the submitted integer results and the Merkle root, then transitions the election to the Tallied state.

The circuit binds correctness at the level of curve points, not integers. Per-candidate integer-to-point correspondence ($n_j \cdot G = R_j$) is verifiable by any auditor who recomputes the scalar multiplication, but is not enforced by contract constraints. This design boundary is discussed further in Section~5.8.

\section{Merkle Audit Construction}

The Merkle audit tree provides a compact digest of all ballot commitments, enabling logarithmic-time inclusion verification for individual voters. The tree takes as input the ordered sequence of $V$ on-chain commitments and produces a single root hash that is stored on-chain after tally finalisation.

Leaf nodes are the commitment values themselves, with no additional hashing:
\begin{equation}
\mathit{leaf}_i = \mathit{commitment}_i
\end{equation}

Internal nodes are computed using Keccak256 over the concatenation of their two children:
\begin{equation}
\mathit{node} = \mathsf{Keccak256}(\mathit{left} \parallel \mathit{right})
\end{equation}

Leaf values are Poseidon commitments produced by the vote circuit, while internal nodes use Keccak256 for EVM-native compatibility in the on-chain audit tooling.

The Merkle tree construction rests on three deterministic rules.

Commitments are sorted first---by \texttt{blockNumber}, then \texttt{transactionIndex}, then \texttt{logIndex}. This produces a canonical order. Any implementation fed the same events will compute exactly the same root.

Odd layers require a quick fix. When a layer holds an uneven node count, the final node is duplicated to act as its own sibling. Pairing then completes cleanly.

Root immutability is locked in without compromise. Once the election enters the Tallied state, a state guard in the contract blocks every further \texttt{merkleRoot} update. The audit digest becomes fixed on-chain, unchangeable.

Proving any single commitment's inclusion needs only $O(\log V)$ sibling hashes along the leaf-to-root path. Voters combine this proof with their local receipt. Section~3.9 sets out the exact verification steps.

\section{Anonymous Verification Procedure}

This procedure allows a voter to confirm that their ballot commitment was included in the audited set, without revealing their identity or ballot content during the verification step. The voter retains a local receipt containing five fields: candidate ID, salt, commitment, transaction hash, and election ID.

Verification proceeds as follows:
\begin{enumerate}
    \item recompute commitment from candidate ID and salt;
    \item locate the entry matching both commitment and transaction hash in the audit bundle;
    \item verify the Merkle path from the matched leaf to the bundle root;
    \item accept if and only if the path is valid, the bundle root equals the on-chain \texttt{merkleRoot}, and the election metadata matches.
\end{enumerate}

Because the procedure relies on commitment and transaction hash rather than the voter's address, no address-based lookup is required. However, on-chain vote events still record the sender address alongside the commitment; this linkage is visible at the chain layer and falls outside the scope of this verification-flow abstraction.
