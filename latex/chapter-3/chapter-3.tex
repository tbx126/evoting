\section{Roles, Assets, and Trust Boundaries}

The protocol defines three active roles: administrator, registered voter, and public verifier. The main protected assets are ballot confidentiality, result correctness, and audit trace integrity.
A public verifier is any party that can read blockchain data without credentials: they can observe on-chain state transitions, inspect proof-verification outcomes, and run the audit procedure in Section 3.9 using the published audit bundle.

Trust boundaries are split as follows:
\begin{itemize}
    \item \textbf{On-chain boundary}: election state transitions and proof-verification outcomes are publicly observable and tamper-evident.
    \item \textbf{Client boundary}: ballot construction, witness generation, and receipt storage occur in user-controlled runtime environments.
    \item \textbf{Administrator boundary}: tally decryption key custody, result submission, and voter-registration whitelist administration remain operationally sensitive; whitelist correctness is controlled unilaterally and cannot be independently verified from on-chain data alone.
\end{itemize}

\section{Threat Model}

The model considers malicious voters and passive blockchain observers. For administrator behavior, the prototype assumes a \emph{semi-trusted administrator}: the administrator is expected to follow the protocol, while single-key control remains an explicit trust assumption rather than a cryptographically enforced guarantee. The following attacks are explicitly targeted:
\begin{itemize}
    \item invalid ballot shape (non-one-hot encryption);
    \item out-of-range candidate encoding;
    \item replay or duplicate voting attempts;
    \item fabricated tally claims unsupported by valid proof.
\end{itemize}

The following are not fully solved in the current prototype: coercion resistance, malware-compromised browser clients, threshold key governance, fully adversarial administrator behavior (for example refusing to tally or submitting inconsistent off-chain artifacts), and per-candidate integer-to-result-point binding at contract layer (ZKP3 constrains curve-point correctness, while integer mapping remains client-verifiable rather than contract-enforced). \section{Protocol State Machine}

Election lifecycle states are:
\begin{equation}
S = \{\texttt{Created},\ \texttt{Active},\ \texttt{Ended},\ \texttt{Tallied}\}
\end{equation}

Allowed transitions are:
\begin{equation}
\texttt{Created} \rightarrow \texttt{Active} \rightarrow \texttt{Ended} \rightarrow \texttt{Tallied}
\end{equation}

This deterministic state machine ensures that vote submission only occurs in the active phase and tally finalization only occurs after the election is ended.

\section{Ballot Encoding and Encryption}

Let $N$ be the number of candidates and $\mathit{candidateId} \in \{0,1,\dots,N-1\}$ be the selected candidate index. A ballot is encoded as one-hot vector
$\mathbf{m}=(m_0,\dots,m_{N-1})$ where $m_{\mathit{candidateId}}=1$ and all other entries are $0$.

For each candidate position $j$, ElGamal ciphertext is generated on BabyJubJub:
\begin{equation}
C1_j = r_j \cdot G, \quad C2_j = m_j \cdot G + r_j \cdot \mathit{PK}
\end{equation}
where $r_j$ is sampled uniformly from the BabyJubJub scalar field, $G$ is generator, and $\mathit{PK}$ is administrator public key. We write $C1_j=(\mathit{c1X}_j,\mathit{c1Y}_j)$ and $C2_j=(\mathit{c2X}_j,\mathit{c2Y}_j)$ for affine coordinates, which are used by the vote-proof witness and ciphertext binding hash. The notation is general in $N$, while the current deployed prototype instantiates $N=2$ and enforces this cardinality at contract layer.

\section{Commitment and Ciphertext Binding}

For private random salt $\mathit{salt}$, commitment is defined as:
\begin{equation}
\mathit{commitment} = \mathsf{Poseidon}(\mathit{candidateId},\, \mathit{salt})
\end{equation}

Ciphertext binding hash is defined over all ciphertext coordinates:
\begin{equation}
\begin{split}
H_{ct} = \mathsf{Poseidon}(&\mathit{c1X}_0,\,\mathit{c1Y}_0,\,\mathit{c2X}_0,\,\mathit{c2Y}_0,\,\dots,\\
                  &\mathit{c1X}_{N-1},\,\mathit{c1Y}_{N-1},\,\mathit{c2X}_{N-1},\,\mathit{c2Y}_{N-1})
\end{split}
\end{equation}

The vote proof uses $(\mathit{commitment}, H_{ct}, PK_x, PK_y)$ as public signals and proves consistency between commitment, candidate index, and encrypted one-hot structure.
Here, $PK_x, PK_y$ denote affine coordinates of $\mathit{PK}$.
In the reference tally client, each emitted ciphertext payload is re-hashed off-chain and matched against the on-chain vote record $(\mathit{commitment}, \mathit{ciphertextHash})$ before aggregation; mismatched entries are excluded from the tally set. This is a client-side integrity requirement rather than a contract-enforced invariant.

\section{Vote Proof Constraints}

The vote circuit enforces four constraint families:
\begin{enumerate}
    \item \textbf{Commitment consistency}: public commitment equals Poseidon of private candidate index and salt.
    \item \textbf{Range check}: candidate index is strictly less than configured candidate cardinality.
    \item \textbf{One-hot encryption validity}: encrypted slot equals 1 for selected index and 0 otherwise.
    \item \textbf{Ciphertext binding}: public ciphertext hash equals Poseidon hash of all ciphertext components.
\end{enumerate}

This composition ensures that an accepted vote is both syntactically and semantically valid under protocol rules.

\section{Homomorphic Tally and Tally Proof}

For each candidate position $j$, aggregated ciphertext is computed by point addition across all votes:
\begin{equation}
\bar{C1}_j = \sum_{k=1}^{V} C1_{k,j}, \quad \bar{C2}_j = \sum_{k=1}^{V} C2_{k,j}
\end{equation}
For tally-proof public signals, we write $\bar{C1}_j=(\bar{C1}^{X}_j,\bar{C1}^{Y}_j)$ and $\bar{C2}_j=(\bar{C2}^{X}_j,\bar{C2}^{Y}_j)$ in affine form.

Given secret key $\mathit{sk}$, decryption result point is:
\begin{equation}
R_j = \bar{C2}_j - \mathit{sk} \cdot \bar{C1}_j
\end{equation}

Each $R_j$ is a curve point of the form $R_j = n_j \cdot G$, where $n_j$ is the integer vote count for candidate $j$. The integer $n_j$ is recovered off-chain by bounded discrete-log solving (baby-step giant-step) and then submitted to the contract as tally integers.

The tally circuit proves:
\begin{itemize}
    \item key ownership consistency: $\mathit{PK} = \mathit{sk} \cdot G$;
    \item decryption correctness per candidate;
    \item sum consistency of all result points with total vote count.
\end{itemize}

The contract also checks that submitted $\mathit{totalVotes}$ matches its internal on-chain vote counter before accepting the proof, bridging circuit-level constraints and ledger state.
In the current prototype, ZKP3 binds correctness of result \emph{points}. The contract writes these result points into on-chain state and emits them in a tally event so auditors can verify consistency between published integers and point-form tally evidence; however, per-candidate integer-to-point binding is not enforced by contract constraints.

\section{Merkle Audit Construction}

The audit set uses commitment values as leaves directly:
\begin{equation}
\mathit{leaf}_i = \mathit{commitment}_i
\end{equation}

Internal nodes are built using Keccak256 over ordered child pair:
\begin{equation}
\mathit{node} = \mathsf{Keccak256}(\mathit{left} \parallel \mathit{right})
\end{equation}

Leaf values remain Poseidon commitments already produced by the vote circuit, while internal nodes use Keccak256 for EVM-native compatibility in audit tooling.
If a layer has odd cardinality, the last node is duplicated. Commitments are sorted by
\texttt{blockNumber}, \texttt{transactionIndex}, and \texttt{logIndex} before tree construction; this yields deterministic roots provided all implementations apply the same ordering.
After tally finalization, the contract rejects \texttt{merkleRoot} updates through a Tallied-state guard, preserving the audit digest as immutable state.

\section{Anonymous Verification Procedure}

A voter stores local receipt fields: candidate ID, salt, commitment, transaction hash, and election ID.
Verification executes:
\begin{enumerate}
    \item recompute commitment from candidate ID and salt;
    \item locate the entry matching both commitment and transaction hash in the audit bundle;
    \item verify Merkle path to bundle root;
    \item accept if and only if proof is valid, bundle root equals on-chain \texttt{merkleRoot}, and election metadata matches.
\end{enumerate}

This procedure allows individual inclusion checks without address-based lookup in the verifier flow. Address-to-commitment linkage in public events remains visible at chain layer and is outside this verification-flow abstraction.

\section{Complexity and Scalability Considerations}

For $V$ votes and $N$ candidates:
\begin{itemize}
    \item ballot encryption complexity is $O(N)$ per voter;
    \item vote-proof generation is the dominant per-voter latency (measured at 3--8 seconds in the current circuit scale) and is not captured by the $O(N)$ encryption term;
    \item tally aggregation complexity is $O(VN)$ point additions;
    \item single inclusion proof verification is $O(\log V)$ hash operations.
\end{itemize}

While the current implementation is suitable for small-to-medium election sizes in prototype settings, larger deployments require optimization of proof generation latency, event indexing, and key governance.
