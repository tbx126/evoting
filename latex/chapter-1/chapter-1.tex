\section{Backgrounds}

Digital transformation has been integrated into many different aspects of the community. Electronic voting systems have features of convenience and efficiency. For decades, these strengths have kept digital technology everlasting prosperity in related reform discussion.

Yet, turning these features into practice is far more difficult than expected.

Ballot secrecy, transparency for verification, and practical feasibility are three significant requirements for every functional e-voting system to satisfy. Meeting those competing demands at the same time always introduces tensions and imbalances, which necessitating trade-offs.

Centralized systems provide users with smooth experience and fast processing speed. To do so, voters have to concentrate trust on server operators. Those operators control logs, manage state changes, and release the final counting results. Independent checks reach only as far as the managers choose to allow. When disputes arise, investigators face restricted data access, making it impossible to conduct independent verification.

This dissertation pursues a different approach. It shifts voters' trust from electoral operators to mathematics and cryptographic algorithms.

ElGamal homomorphic encryption, Groth16 zero-knowledge proofs and commitment-based Merkle inclusion proofs are integrated together to address privacy protection, verification transparency and ballot inclusion check.

This prototype goes beyond a purely theoretical outline. It compiles successfully, support deployment on local environment and real Ethereum test network (Sepolia). The system design has undergone automatic testing and is available for independent review.


\section{Motivations}

Three convergent technical advances make the proposed combination of methods practically feasible.

First, programmable blockchain such as Ethereum provide Solidity environment for smart contracts with append-only event logs and deterministic execution codes. These properties eliminate an entire class of attacks common to centralized systems---namely, silent, undetectable modifications to election records \cite{nakamoto2008,wood2014}.

Second, with zk-SNARK constructions, especially in the Groth16 proving system, succinct proofs can be easily generated and the on-chain verification cost remain acceptable within the gas limit \cite{groth2016}.

Third, compared to traditional hash function, such as SHA-256, Poseidon is optimized for arithmetic circuits and become much more algebraically efficient. When it's combined with Merkle tree constructions, vote inclusion checks over larger ballot sets can be practical. This makes the system remains scalable and cost-effective, since the cost increases logarithmically, rather than linearly, with the total number of ballots \cite{poseidon2021,merkle1988}.

To build on these advances, this study has these five objectives:

\begin{itemize}
    \item Create a state machine that enforces explicit lifecycle transition for the election.
    \item One-hot ElGamal encryption on the BabyJubJub curve is adopted to ensure that every ballot is valid and encrypted.
    \item Use Groth16 zero-knowledge proofs to verify the validity of individual ballot and the correctness of tally derived from off-chain homomorphic aggregation.
    \item Develop a receipt-based audit mechanism whereby voters can confirm, using a locally stored receipt and a published audit bundle, that their ballot commitment has been correctly included.
    \item Evaluate the strengths of the resulting implementation alongside its limitations, including residual engineering constraints, trust assumptions, and binding gaps that currently prevent higher-assurance deployment.
\end{itemize}

\section{Objectives and Scope}

This e-voting system is a reproducible research prototype, targeting deployment of a two-candidate election.

Circuits are compiled and tested for a fixed candidate count of $N = 2$; larger $N$ requires significantly more engineering effort and is not part of this prototype. Voter eligibility relies on an administrator-managed whitelist whose real-world correctness is outside the protocol's cryptographic scope.

\textbf{Trust assumptions.} The prototype assumes a semi-trusted administrator who follows the protocol. The admin keeps the private key safely, gives an honest whitelist, and would not release wrong results since it can be easily identified by the public. For on-chain results, people only need to trust cryptography and no institutional trust is needed.

Out of scope: coercion resistance; threshold key management; decentralized governance; formal contract verification; cross-election voter privacy.

This work produces four concrete contributions:

\begin{itemize}
    \item A cross-stack prototype spanning Solidity smart contracts, Circom zero-knowledge circuits, browser-side JavaScript for cryptographic computation and wallet interaction, and Python for local serving and cryptographic validation. The prototype compiles, deploys to Ethereum-compatible networks, and is accompanied by 76 automated tests covering both contract behaviour and underlying cryptographic primitives.
    \item A vote-submission mechanism that cryptographically binds each ballot's Poseidon commitment to the corresponding encrypted payload through a shared ciphertext-binding hash, both of which serve as public inputs to the vote proof. This binding reduces the risk of discrepancies between the statement proved by the zero-knowledge proof and the data recorded on-chain.
    \item A tally pipeline that performs homomorphic ciphertext aggregation off-chain, verifies decryption correctness through an on-chain Groth16 proof, and anchors the finalised result set with a published Merkle root of all ballot commitments.
    \item An audit path that enables every voter to confirm whether their ballot is included using a receipt and an audit bundle which provided by the admin. In this process, the voter's on-chain address is unnecessary and the verification procedure checks commitment inclusion via a Merkle tree and validates root consistency with the on-chain root hash value.
\end{itemize}
